# FocusDo 产品分析

## 零、产品哲学

### 核心理念：扁平 Projects，强制掌控

FocusDo 采用**扁平的项目结构**，而非嵌套的层级目录。

**设计原则**：
- 每个项目列就是一个独立的"关注域"（投资、工作、生活、学习...）
- 用户必须自己决定"这件事属于哪个维度" — 这个决策过程本身就是一种掌控感
- 避免了"扔进大杂烩就忘了"的问题

**为什么不嵌套**：
- 嵌套增加认知负担 — "这个任务到底该放哪个子目录？"会让人纠结
- 真实使用场景中，大多数人的待办事项就是 3-8 个维度，扁平结构完全够用
- 添加新项目的成本很低，用户可以自由调整粒度

**用户自由度**：
- 想列 5 个项目就 5 个，想列 20 个也行
- 每个用户可以根据自己的生活/工作节奏定义项目维度
- 这是 FocusDo 区别于其他 Todo 工具的核心差异

## 二、致命问题（不修就不能用）

### 1. ~~零持久化 —— 刷新即失~~ ✅ DONE
~~所有数据在内存里，刷新页面回到初始状态。~~ 已通过 localStorage 持久化解决。

### 2. 没有编辑能力
创建之后就不能改了。发现打错字？只能删了重建。没有编辑文本、改优先级、改日期的功能。

### 3. 没有 undo / 删除确认
点了删除就没了，没有二次确认，没有撤销提示。误删 = 数据永久丢失。

---

## 三、体验短板（用了会觉得不好用）

### 4. 没有搜索
当 todos 多了之后，找特定事项只能靠眼睛一个个扫。至少需要一个文本搜索。

### 5. ~~列不能自定义~~ ✅ DONE
~~三列（投资/工作/生活）是写死的。~~ 已支持新增自定义项目列。

### 6. 没有排序
todos 的顺序是创建顺序。不能按优先级、截止日期、完成状态排序。

### 7. 筛选交互待优化
每个 project 列都有独立的筛选 Tab（全部/进行中/今日/已完成），UI 上显挤，但功能上又需要。需要一个更优雅的筛选方案，可能是全局筛选 + 列内简化，或者把筛选移到其他位置。

---

## 四、缺失的差异化功能（做了就能拉开差距）

### 8. 拖拽排序 / 跨列移动
看板类产品没有拖拽是不完整的。至少支持同列内拖拽排序，最好支持跨列移动。

### 9. 每日回顾 / 周报生成
既然有"今日焦点"的概念，应该有"今日回顾"——自动汇总今天完成了什么、逾期了什么，生成一段文字摘要。这是 FocusDo 跟普通 Todo 最大的差异点。

### 10. 数据统计仪表盘
环形进度只在列头有一个小的。可以做一个汇总页：本周完成率、各项目对比、逾期趋势、完成热力图。

### 11. 快捷键支持
`N` 新建、`/` 搜索、`1/2/3` 切换项目列、`Enter` 确认、`Esc` 取消——对效率用户是刚需。

### 12. 暗色模式
现在是暖色纸张风格，暗色模式需要单独适配。

---

## 五、项目管理增强（待讨论）

### 13. 项目支持编辑
用户创建项目后，可以修改项目名称、颜色、图标。

### 14. 项目支持调整排序
当项目列较多时，用户可以拖拽调整项目列的顺序，方便将重要的项目放在前面。

### 15. 项目支持归档
项目完成后可以归档，归档后的项目不再显示在主视图中，但数据保留。适用于阶段性项目（如"双十一活动"结束后归档）。

### 16. ~~项目是否支持删除~~ ✅ DONE（决策：只保留归档）
**决策**：现阶段只用归档来实现"删除"的功能，不单独提供删除按钮。

**理由**：
- 归档 = 不再显示，但数据保留，可恢复，效果等同于删除
- 避免用户纠结"该归档还是删除"
- 避免误操作导致数据丢失

**后期扩展**：如果用户确实需要永久删除单个项目，可以在归档列表里增加删除功能，而不是在 Projects 主界面增加删除按钮。

## 七、技术债务

- 所有样式使用 inline style，维护成本高，后续建议迁移到 CSS Modules 或 Tailwind
- 数据类型定义在组件文件内，应抽离到独立的 types 文件
- 没有单元测试和 E2E 测试
- `nextId` 使用模块级变量，多实例不安全，应迁移到 UUID 或 crypto.randomUUID()
